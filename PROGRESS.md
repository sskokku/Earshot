## Speaker curation (needs-naming queue + merge suggestions) (2026-06-14) ✅
- **Scope:** Out-of-band post-v1 feature. A dedicated "Needs Naming" window (`Cmd+Shift+N`) surfaces every active speaker without a user-assigned name, ranked DESC by total speaking frequency (segment count) so the user works through the most-impactful labels first. Each row shows the speaker's persistent `display_label`, segment count, per-context embedding counts, and up to three sample quotes (longest text first, tie-broken by most recent) pulled from the `segments` table to aid voice recognition without scrubbing the transcript. Inline TextField + Save button per row runs through the existing transactional `SpeakerLibrary.renameSpeaker` (UPDATE speakers + Markdown rewrite inside a single `dbQueue.write`) plus a resolver-invalidate so the next utterance picks up the new label without a pipeline restart. Below the unnamed list, a "Merge suggestions" section surfaces pairs of active speakers whose cross-context cosine similarity sits inside `[0.60, 0.75)` — just below the live resolver's auto-merge ceiling — with one-click confirm-merge that runs `SpeakerLibrary.mergeSpeakers` (named > more-segments > lower-id preference picks the destination). The menu's top status line shows a textual "· N to name" badge when the count is non-zero, and a "Needs Naming (N)…" menu item carries the same count. No forced modal; this is a view the user visits when they want to clean up the library. The badge is refreshed after every library mutation and on a 30 s timer so a newly-minted speaker (from the live resolver mid-call) lights up the menu without manual intervention.

- **Decisions:**
  - **Suggestion band is `[0.60, 0.75)`, cross-context only.** The live resolver auto-folds at `>= 0.75` cross-context; anything in `[0.60, 0.75)` is "almost-but-not-quite" — the cases most worth a human glance. Below 0.60 the false-positive rate gets ugly fast. Same-context near-matches aren't surfaced because same-context already auto-folds at 0.65; surfacing `< 0.65` same-context pairs would mostly be noise.
  - **Destination preference: named > more segments > lower id.** A named speaker should always absorb an unnamed one (so the human-readable label survives). Among equal-quality candidates, the one with more accumulated evidence wins. Lower-id is the deterministic last-resort tie-break for tests.
  - **Sample quotes are length-first, recency-tie-broken.** A 30-word utterance is meaningfully more recognizable than a 3-word one; among similar-length quotes, fresher is better since the user's memory of recent conversations is stronger. The query is `ORDER BY LENGTH(text) DESC, start_ts DESC LIMIT N`.
  - **Inline rename, not a sheet.** The spec asked for "inline name assignment" — having a TextField + Save button right on the row matches the affordance described. The library window's per-row "Rename…" sheet remains for renames done from the library surface (where the user wanted a clear focused dialog); the curation surface is optimized for many quick renames in sequence.
  - **Badge is textual, not an icon overlay.** First attempt overlayed a red `Capsule + Text` on the menu bar `Image`. `MenuBarExtra` with `.menuBarExtraStyle(.menu)` expects a stable image-shaped label closure — a custom View wrapper with overlay sent the system into a layout loop that beachballed the app at boot. Reverted to a plain Image and surfaced the count textually in the menu's top status line ("EarShot — Listening · 3 to name") and on the "Needs Naming (3)…" item. Both are visible the moment the user clicks the menu bar, which is when the badge matters.
  - **30 s badge refresh cadence + after-mutation refresh.** Long enough that the periodic COUNT(*) never costs anything noticeable; short enough that a new voice in a meeting lights up the badge before the user looks at the menu. The after-mutation hook closes the gap entirely for rename / merge actions in either curation window or the speaker library window.
  - **`AppSettings.ownerSpeakerIDValue = nil` at the top of each curation test.** UserDefaults persists across the whole xctest run, so a prior suite that enrolled an owner left a non-nil id behind. The curation queries filter by `id <> ownerSpeakerIDValue` — a stale owner id excluded one of the test speakers and was the first test failure I hit. The reset is explicit per-test rather than in a `@Suite.serialized` init because the prior tests don't centralize UserDefaults setup either.
  - **`vectorWithCosine(to:cosine:)` test helper, not seed blending.** First attempt blended two seeded vectors with alpha=0.7 expecting cosine ≈ 0.68. The actual cosine for near-orthogonal seeds at alpha=0.7 is ≈0.92 (above the 0.75 ceiling) — and the test failed because the in-band pair never appeared. Replaced with a Gram-Schmidt construction that produces a unit vector with EXACT target cosine to a reference. The math is more code but the test now expresses intent ("0.68 cosine") instead of an approximation that depends on the seeds.
  - **Test-only `testForceName` / `testForceMergedInto` on the actor.** Curation tests need to flip `name` / `merged_into` without running the full `renameSpeaker` / `mergeSpeakers` paths (those expect a transcript folder + writer). The actor-internal direct-SQL setters are the cleanest escape hatch; the test helper extension wraps them so call sites stay readable.
- **Exit-test status:** ✅ project builds clean. All 6 new tests pass individually (`unnamedSpeakerCountExcludesOwnerMergedAndNamed`, `unnamedSpeakersRankedByTotalSegmentFrequency`, `sampleQuotesPrefersLongestAndMostRecent`, `mergeSuggestionsBandPassesAndRanksDescending`, `mergeSuggestionRecommendsNamedDestination`, `recommendMergeDirectionTieBreaksByLowerID`). Prior suite spot-check: `SpeakerLibraryS4Tests/renameSpeakerUpdatesNameAndDisplayLabel` still passes against the augmented library (test-only backdoors did not regress production code). Live verification (open with Cmd+Shift+N, name a couple voices, confirm transcript relabels, confirm a merge suggestion folds two unnamed rows into one, confirm the badge count drops and the menu top line + menu item both update) is a human-only step.

## Timeline view + curation (2026-06-13) ✅
- **Scope:** Out-of-band post-v1 feature. Renders the `sessions` + `bookmarks` data model the prior chunk stood up as a horizontal day timeline (`Cmd+Shift+T`) — one block per session, colored by source (mic = teal, system = orange, both = purple), labeled with the session's text label (or `Ambient`/`Call` + first bookmark when no label is set), positioned by `started_at`/`ended_at` against a 00:00–24:00 axis with 2h ticks. Right-click a block → context menu with "Open transcript" + "Redact this session…" pre-filling the redaction sheet with the block's bounds. The "Redact Range…" button in the header opens the same sheet pre-filled with the full visible day. - **Decisions:**
  - **One-day-at-a-time view.** A multi-day timeline would either need a horizontal scroll surface or a vertical-stacked-days layout; both add UX surface area for marginal value. Most ambient days have a small handful of sessions; the day-picker + arrow-keys flow lets the user scrub across days quickly without a more complex view.
  - **Click → reader window, not in-window detail pane.** The reader window already exists and is the right surface for rendering a transcript with a focused line. Adding a side-pane detail view to the timeline would duplicate the reader's responsibilities and require a second focus implementation. The cost of a second window is one click; the win is no duplication.
  - **Block label fallback order: explicit `label` → "Call" / "Ambient" + first bookmark → just "Call" / "Ambient".** Surfaces the most-meaningful label the user has supplied (an explicit session label is the strongest signal; a bookmark dropped during the session is weaker but still meaningful; the kind alone is the floor). Keeps the timeline scannable even before any naming has happened.
  - **Redaction is permanent, not soft-delete.** A soft-delete column would let "undo" work but would leak: search queries would have to filter `WHERE NOT redacted`, the FTS index would still carry the text, and a curious user could SELECT * the deleted rows back. The user's requirement here is `permanently and transactionally` — soft-delete contradicts both halves.
  - **Bookmarks captured inside the redaction window are deleted too.** A bookmark labels a moment in the conversation; if the conversation at that moment is gone, the label has no referent. Leaving the bookmark would create a dangling label that points at empty space.
  - **Sessions rows are NOT deleted by redaction.** Different judgment: the session row encodes "a stretch of listening happened from X to Y" — that fact survives the scrub of what was said. The block stays the same width until the next reload, which is the correct behavior — the timeline is a snapshot, not a live ticker.

All 78 unit tests enabled, the new 10 (TimelineQueryTests + RedactionTests) pass on first run, the prior 26 SpeakerLibrary / Sessions / Bookmark tests still pass.   - **Schema lives in the SpeakerLibrary SQLite file, not a new DB.** Earshot's "single library" already holds speakers, embeddings, segments, FTS5 search log.   - **Backfill via raw SQL inside the v4 migration, with a public `backfillSessionsFromSegments` method for tests and a future admin tool.** Inline-migration backfill is the textbook GRDB pattern; exposing the same INSERT through an instance method lets tests verify behavior against arbitrary segment populations without having to manually un-run and re-run migrations.

The window has room; widening the default keeps the same call sites working (existing tests still pass) and gives the search UX more to chew on.   - **Crash recovery is a marker, not a repair.** Rule 4 already gives us append-only writes with fsync per line, so the on-disk file survives a crash by construction. Repair logic (rebuild from segments table, replay missing lines) would only matter if writes weren't fsynced, and they are.
  - **Lifecycle flag in UserDefaults, not on disk.** A separate flag file would need its own atomic-write story; UserDefaults is already thread-safe and survives a crash. Cleared in `cleanupForTermination`, set in `applicationDidFinishLaunching`. A SIGKILL leaves it set; the next launch sees it and emits the marker.
  - **Bucket crash recovery as `routeChange` recovery.** CLAUDE.md's error taxonomy is intentionally tight (6 classes). "Crash recovery" is functionally "we lost capture and rebuilt" which maps to `routeChange`. Adding a 7th class would force every emitter to think about it for one annual event. The human-readable signal lives in the marker + the disk log line.
  - **5-min metrics flush, not on-every-event.** Per-event flush would serialize the live VAD/ASR loop behind disk; the 30 s sampler hop already exists and 5 min is a sensible loss window for the daily JSON sidecar. Trades up to 5 min of counter loss in a crash for a fast live path.
  - **SoakHarness is a side-channel observer + synthetic audio injector, not a pipeline replacement.** It does not bypass `AVAudioEngine` or the Core Audio tap; it pushes additional samples into the existing AsyncStream alongside whatever real audio is flowing. This means a soak run on a noisy desk also exercises the live audio path. The downside is that real audio + synthetic audio interleave; for a clean signal-only soak, run on a quiet desk.
  - **Synthetic audio is sine + envelope, not formant-modeled speech.** The goal is exercising the VAD → ASR → diarizer → merge → SQLite → file writer chain at a controllable cadence; transcription accuracy isn't the metric (sine waves aren't speech). For 12-hour memory-stability validation the input doesn't need to be Parakeet-friendly.
  - **30-day log retention.** Same shape as the PRD R5 transcript retention setting (30/90 days). The disk log is small enough (one line per error event, typically <1 KB per day on a normal run) that this is generous. Pruning runs on boot + on every date rollover.
  - **Launch-arg activation, not menu item.** The soak harness produces no user value during normal operation. Surfacing it as a debug menu item would just clutter the menu. `-EarShotSoakMode YES` is the right cost surface for a developer running a verification soak.
  - **`SoakHarness.tickOnce` is public for testing.** Without it the test would have to wait one full `sampleInterval` for the background loop to fire; making it public is cheap and the cost (someone calling it outside tests) is negligible since the harness is debug-only.

  - Survival checklist:
    - App Nap defeated: in `~/Earshot/logs/`, the soak log's resident memory column shouldn't dip during long idle periods (App Nap throttling would otherwise cause measurable inactivity).
    - Sleep assertion on AC only: unplug the Mac while listening; `~/Library/Caches/com.earshot.app` console logs show `SleepAssertion: release()` immediately. Re-plug; assertion reacquires.

    - LaunchAgent at login: open Settings → check "Launch at login", quit, log out, log back in. EarShot should auto-launch.

### Chunk CP1 — Offline correction pass (every 5 min, atomic Markdown rewrite, silent live updates) ✅
- **Scope:** PRD R6 lands. Every 5 minutes, an offline re-diarization runs against the trailing 5 minutes of audio for each pipeline; provisional segments are re-resolved through `IdentityResolver` using the offline pass's per-slot embeddings (so improvements in the library since the segment was first written — named speakers, merged identities, accumulated cross-context evidence — flow back into the on-disk transcript). Thermal pressure (`.serious` / `.critical`) flips a throttle so the pass becomes a no-op until the system cools. Relabel counts roll into the daily metrics summary per PRD R8.
- **Files / types:**
  - `EarShot/CorrectionPass.swift` (new). `actor CorrectionPass` owns the 5-min cadence (`Task.sleep` loop), a dedicated `DiarizerManager` (~250 MB) so `performCompleteDiarization` calls don't pollute the shared `EmbeddingExtractor`'s manager state, and references to `SpeakerLibrary`, `IdentityResolver`, `TranscriptWriter`, `MetricsCollector`, and `LiveTranscript`. `runOnce()` is public so a future debug menu item can trigger an on-demand pass; the per-pipeline `runPass(context:snapshot:)` is the core. `dominantSlot(span:segments:)` is the overlap-weighted vote that picks each existing DB segment's new offline slot. `CorrectionAudioSnapshot` is the Sendable carrier the pipelines hand back. `setThermalThrottle(_:)` is the pause hook the `ThermalMonitor` drives.

CLAUDE.md rule 2 ("rolling 30 s buffer max") gets a deliberate exception because rule 5 ("audio is ephemeral, never to disk") forces the offline pass to keep its working window in RAM — there is no other source. At 16 kHz mono Float32 the buffer is ~19 MB per pipeline, ~38 MB total, which is acceptable for the always-on Mac this app targets. `appendToCorrectionBuffer(_:)` trims oldest samples and advances `correctionBufferEpoch` so the slice math in `CorrectionPass` lines up with the segments table's wall-clock timestamps. - **Architecture-rule check:**
  - Rule 2 (nothing accumulates in RAM): ⚠️ **deliberate exception** — each pipeline now carries a 5-min PCM ring (~19 MB). Justified by R6 + rule 5: the correction pass needs audio, audio cannot live on disk, so it lives in RAM for a bounded window. Both pipelines clear the ring on start/pause; the cap is enforced by `removeFirst(overflow)` so growth is impossible. Future-flag: if memory budget gets tight on M-series, the buffer compresses well (mono Float32 → Float16 is lossless for downstream WeSpeaker since the model L2-normalizes).
  - Rule 3 (provisional then corrected): ✅ the pass is the only writer that flips `provisional` to 0. The live path never blocks on it — it runs on its own actor with a `Task.sleep` cadence.
  - Rule 4 (transcript file is the product): ✅ atomic rewrite via temp file + `FileManager.replaceItemAt` inside the same `dbQueue.write` block that updates the SQL rows; rewrite failure → DB rollback.
  - Rule 5 (audio is ephemeral): ✅ audio still never touches disk; the 5-min ring is RAM-resident and reset on pause.
  - Rule 6 (fail quiet, recover loud-free): ✅ `performCompleteDiarization` failures bucket as `.diarizerFailure` + a recovery attempt; `segmentsInRange` failures bucket as `.diskWriteFailure`; `applyCorrections` failure (per-day) bucket as `.diskWriteFailure`. No modal, no glyph flip.
  - Rule 8 (merge layer owns identity): ✅ the correction pass talks to `IdentityResolver` directly (same engine the merge layer drives on the live path); no other component is reached. After each pass, the resolver's cache is invalidated for every touched `speakerID` so the live path picks up new display labels on the very next utterance.

- **Decisions:**
  - **Dedicated `DiarizerManager` for the correction pass.** Shared `EmbeddingExtractor` wraps a `DiarizerManager` already, but `performCompleteDiarization` mutates the internal `SpeakerManager` state across chunks while `extractSpeakerEmbedding` (the extractor's call) does not. Mixing both on the same instance is fragile under concurrency. The 250 MB cost is acceptable for a personal-use always-on tool; `speakerManager.reset()` runs at the top of every pass so slot numbering doesn't accumulate across days.
  - **Per-slot embedding, not per-segment.** The offline result's `TimedSpeakerSegment` carries a 256-d L2-normalized embedding per output segment. We pick the longest-duration sample per slot as the slot's representative embedding and reuse it across every DB row that the slot dominates. This gives all segments in the same offline cluster a consistent persistent identity in one pass, which is the win over re-resolving each DB segment independently (otherwise we'd just be re-running S3 with a more expensive embedding extractor).
  - **Match by (source, startedAt, text), not by UUID.** The DB's `segments` table doesn't carry the live `LiveTranscript.Segment.id` UUID (segments are inserted from the merge layer's forwarder, which doesn't have a stable ID across the panel boundary). The same tuple used by S4's `applyRelabel` is reused here; idempotent if a pass re-runs the same correction.
  - **Fresh `sessionID` per pass + `slotLabel = "offline:<slot>"`.** The resolver's per-(source, session, slot) cache would otherwise short-circuit on a stale identity from the live path's pipeline session. A pass-local UUID keeps the resolver doing a real DB lookup for each unique offline slot. Inside the pass we also cache per-slot resolutions so we hit the DB once per slot, not once per segment.
  - **First pass waits one full interval.** The boot-time buffer is empty; a pass at t=0 spins against nothing and just bumps zero counters. Waiting one cadence means the first useful pass fires at t=5 min when there's actual audio + segments to work with.
  - **Race window same as S4.** Between `pauseForRelabel` returning and the in-transaction `replaceItemAt`, a fresh `writer.append` could land on the freshly-reopened handle, then be lost when the swap happens. Same window S4 carries. Accepted here because the pass fires once every 5 minutes and the window is milliseconds; if it bites in soak testing, the fix is to add an explicit "writer paused" gate the writer queues appends behind.
  - **No new menu surface.** The pass is fully invisible — by design (rule 3). Console logs (subsystem `com.earshot.app` category `CorrectionPass`) surface every decision: per-pass candidate counts, offline segment counts, relabel counts, errors, thermal throttle entry/exit.
- **Exit-test status:** ✅ 47 unit tests pass (41 prior + 6 new). Live verification of "induce a known mislabel; confirm the file is corrected within one cycle without UI disruption" is a human-only step per BUILD_PLAN's exit test — needs two similar-sounding voices in a room talking for 5+ minutes.
- **Known limitations / Phase 5 prep:**
  - The correction pass does not change segment **boundaries** in the on-disk file, only labels. The offline diarizer's better boundaries influence which slot a segment is attributed to (overlap-weighted vote), but the segment's VAD-derived `[startedAt, endedAt]` interval stays put. Boundary rewrites would require splitting/merging Markdown lines, which is a larger surgery than the 5-min cadence justifies.
  - Buffer size is fixed at 5 minutes (`CorrectionPass.windowSeconds`). A future tuning knob could widen it for higher accuracy under heavy ANE budget — VBx clustering scales well with more audio context — but the trade is memory + per-pass latency.
  - `OfflineDiarizerManager` (the full VBx pipeline per FluidAudio's offline docs) was considered and skipped for CP1: smaller surface area to land first, and `DiarizerManager.performCompleteDiarization` already gives the "offline > streaming" delta the PRD asks for. Upgrading to `OfflineDiarizerManager` is a drop-in replacement inside `CorrectionPass.runPass` once we want VBx-level accuracy.
