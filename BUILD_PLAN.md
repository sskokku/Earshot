# Earshot Build Plan

Five phases. Each ends with something you can run all day. Do not start a phase until the prior phase's exit test passes.

> **Resequencing note (2026-06-12):** the original swap (mic-only speaker ID before call auto-attach) has been reversed. Call auto-attach now lands first (C1, C2), THEN diarization sub-chunks S1‚ÄìS4 are applied to BOTH pipelines together. Rationale: standing up the system-audio pipeline turned out to be the riskier of the two ‚Äî it depends on Core Audio process taps, entitlements, and Teams' helper-process structure that the diarizer doesn't influence. Proving capture on both pipelines first means S1‚ÄìS4 can be designed against the real two-pipeline merge layer instead of being rewritten when the second pipeline arrives. S1 (mic side, session-local Speaker N) already shipped under the old ordering; it stays in tree and is folded into the new S1 step, which extends the same diarizer instance pattern to the system pipeline.

## Phase 3: Speaker identification (both pipelines)
S1 mic-side already shipped (`387a0f7`); S1 step below extends the same pattern to the system pipeline so the merge layer sees two diarizer streams from day one of this phase.

- [x] **S1** ‚Äî Streaming diarization (FluidAudio Sortformer) on BOTH pipelines.
  - Mic side: ‚úÖ shipped under the old ordering (`387a0f7`). `DiarizerActor` owns a `SortformerDiarizer`, session-local "Speaker N" labels via slot‚Üínumber mapping, reset on pipeline restart.
  - System side: ‚úÖ a second `DiarizerActor` instance runs against the system-audio pipeline with the same session-local labeling and an independent slot map; wired through `SystemAudioPipeline.setDiarizer` + fire-and-forget feed from `processVadChunk` + dominant-slot lookup in `finalizeCurrent`. Slot maps never cross between pipelines (rule 1). All segments minted by either pipeline carry `provisional = true` via `LiveTranscript.Segment.provisional` (rule 3 + schema). No SQLite, no cross-session memory yet ‚Äî that's S2.
- [x] **S2** ‚Äî SQLite (GRDB) speaker library per CLAUDE.md schema (speakers, embeddings, segments). `SpeakerLibrary` actor wraps GRDB `DatabaseQueue` at `~/Library/Application Support/EarShot/speaker_library.sqlite3`; v1 migration creates all three tables with foreign keys. Onboarding's 30 s "Me" capture now writes the owner row + enrollment embedding straight to the DB; a one-shot migration moves any pre-existing Chunk-2 `me_embedding.bin` into the DB on first boot. Both mic and system pipelines record one 256-d WeSpeaker embedding per finalized segment, tagged with the correct context, capped at 10 per speaker per context (lowest-quality eviction). Shared `EmbeddingExtractor` actor wraps a single `DiarizerManager` since extraction is stateless (rule 1 only covers pre-diarization state). Debug menu item "Dump Speaker Library‚Ä¶" surfaces speaker + embedding counts for verification. No matching logic yet ‚Äî storage and enrollment only; S3 brings cosine matching into the merge layer.
- [x] **S3** ‚Äî Identity mapping: the new `IdentityResolver` actor sits behind the merge layer (rule 8 ‚Äî no other component touches identity) and maps chunk-local Sortformer slot labels from BOTH pipelines to persistent SQLite speaker IDs via cosine matching. Same-context threshold 0.65, cross-context fallback 0.75; below both ‚Üí mints a new persistent speaker with a stable `display_label` ("Speaker N"). Owner enrollment embedding makes the owner's voice match from session start on the mic side. Every confident match (cache hit, same-context, or cross-context) appends a fresh embedding under the matched speaker in the caller's context so recognition improves with use and cross-context evidence accumulates. Per-(source, session, slotLabel) cache short-circuits subsequent segments from the same Sortformer slot. Every decision is logged (subsystem `com.earshot.app` category `IdentityResolver`) ‚Äî candidate speaker, similarity score, threshold, outcome ‚Äî so thresholds can be tuned against real voices without rebuilding. SpeakerLibrary gained a v2 migration adding `display_label TEXT` plus `allEmbeddings(context:)`, `createUnnamedSpeaker()`, and `displayLabel(forSpeakerID:)`; the S2 session-scoped `sessionMap`/`ensureSpeaker` path is gone since identity matching replaces it.
- [x] **S4** ‚Äî Naming UX (right-click any segment in the floating panel ‚Üí "Name `Speaker N`‚Ä¶" sheet) + transactional retroactive relabel (`SpeakerLibrary.renameSpeaker` runs `UPDATE speakers SET name=?, display_label=?` and `FileManager.replaceItemAt` inside the same `dbQueue.write` ‚Äî a file-swap failure throws and the SQL UPDATE rolls back, satisfying CLAUDE.md "transactional speaker naming" at the data layer). Manual merge UI in the new `SpeakerLibraryWindow` (`Cmd+Shift+K`) reassigns embeddings + segments + rewrites today's Markdown with the same atomic pattern; speaker rows survive with `merged_into` populated for audit. The same window adds per-speaker actions for clearing embeddings and deleting a speaker, plus a "Re-enroll Me" flow that re-runs the 30 s capture and atomically replaces the owner's mic-context embeddings. FTS5 keyword search uses a porter-tokenized external-content `segments_fts` virtual table synced to `segments` via GRDB triggers; the merge layer's `onForward` writes a row per finalized segment so search has live data. Queries hit `searchSegments` from the floating-panel search button; every query is persisted in `search_log` and bumped into the daily metrics' `searchesRun` counter (PRD R8). `IdentityResolver.invalidate(speakerID:)` is called after every rename / merge / delete / re-enroll so cached cache entries pick up the new label without a pipeline restart.

Echo dedupe in the merge layer (mic vs system within 2 s, normalized token overlap) is part of S3 ‚Äî once persistent identity exists, dedupe can lean on it instead of just text similarity.

Exit test: name two colleagues on Monday. On a Wednesday call with them ‚Äî mic + system pipelines both running ‚Äî they are auto-labeled correctly across both contexts. Own speech labeled from session start. Search for a phrase said yesterday returns the segment. No duplicated lines from echo.

## Phase 4: Correction pass
- [x] **CP1** ‚Äî Every 5 min, offline re-diarization of the trailing window (offline mode is meaningfully more accurate than streaming). Each pipeline holds a rolling 5-min PCM ring buffer (deliberate rule-2 exception so audio never has to touch disk per rule 5) anchored to a wall-clock epoch. `CorrectionPass` actor runs `DiarizerManager.performCompleteDiarization` on each snapshot, groups in-window `segments` rows by dominant offline slot (overlap-weighted), and re-resolves each slot through the existing `IdentityResolver` so the live path's chunk-local Sortformer slots are reconsidered against the library's current state (named speakers, merged embeddings, accumulated cross-context evidence). The same atomic rewrite pattern S4 uses for naming (`pauseForRelabel` + `applyRelabel` + `FileManager.replaceItemAt` inside `dbQueue.write`) writes the corrected labels back into today's `YYYY-MM-DD.md`; relabeled rows have `provisional` cleared. The live panel reflects updates silently via a main-actor hop into `LiveTranscript.applyCorrectionUpdates`. Thermal `.serious`/`.critical` flips a throttle that skips the next pass. `MetricsCollector.recordSegmentsRelabeled` counts every speaker_id change for the daily summary block / JSON sidecar (PRD R8 "Correction pass: segments relabeled").

Exit test: induce a known mislabel live (similar voices); confirm the file is corrected within one cycle without UI disruption.

## Phase 5: Hardening and soak
- [x] **H1** ‚Äî bundled hardening chunk: crash recovery on relaunch (transcript file resumed in-place via the existing `ensureHandle` reopen path; a `recovered HH:MM:SS` marker is appended so a human reader sees where the prior session ended; `AppSettings.runningSessionFlag` is the watchdog set on launch and cleared in `cleanupForTermination`; segments-table entries use fresh wall-clock timestamps so no duplicate rows can arise). Periodic `MetricsCollector.flush()` every 5 min from the existing 30 s sampler so a crash loses at most 5 min of counters. `FileLogger` actor writes one event line per error / recovery / gap marker / crash recovery to `~/Earshot/logs/earshot-YYYY-MM-DD.log` with fsync per line; daily rotation; 30-day retention pruned on boot + on rollover. `SoakHarness` actor (debug-only, activated by `-EarShotSoakMode YES` launch arg or the equivalent UserDefaults key) generates synthetic 3-speaker sine carriers + envelopes and injects them through both pipelines via new `MicPipeline.injectSoakAudio(_:)` / `SystemAudioPipeline.injectSoakAudio(_:)` test surfaces, then writes one tab-separated row every 60 s to `~/Earshot/logs/soak-yyyyMMdd-HHmmss.log` (resident memory MB, process CPU %, thermal state, mic+system segment totals and deltas, echo drops, errors). Route-change torture surface exposed as `MicPipeline.debugTriggerRouteChange()` + `debugCarryCount()` for the unit test that fires 10 reconfigures in a row and asserts no state growth. Survival checklist already wired in Chunks 5 + CP1 ‚Äî re-audited here: App Nap held the whole listening lifetime (`AppNapAssertion`), sleep assertion held only while listening AND on AC (`SleepAssertion` + `PowerSource.isOnAC`), gap marker on battery wake (`SleepWakeMonitor` + `handleDidWake`), thermal `.serious`/`.critical` widens VAD throttle on both pipelines and pauses `CorrectionPass`, launch-at-login via `SMAppService.mainApp` (`LoginItem`).

Exit test: 10-hour soak passes with stable memory and zero lost segments across two forced route changes and one forced kill/relaunch. **v1 feature-complete pending soak.**
